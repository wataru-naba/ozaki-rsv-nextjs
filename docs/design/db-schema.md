# DB設計 (Prismaスキーマ) 設計ドキュメント

対象: `prisma/schema.prisma`
参照要件: `docs/requirements/current-requirements.md`(特に4章「残る未確認事項」)
参照原典コード: 原典CakePHPリポジトリ(クローン済み) `src/Controller/RsvController.php`, `src/Controller/AdminController.php`, `src/Controller/AdminHyugaController.php`, `src/Model/Table/*.php`
参照旧DB定義: `schema_ozdb.sql`

---

## 背景

Angular + CakePHP構成の予約システムをNext.js(Prisma + PostgreSQL)へ再構築するにあたり、旧DBスキーマ(MySQL/MariaDB)をそのまま踏襲せず、原典コードの実装事実に基づいて型・制約・テーブル構成を見直す。特に旧スキーマで `int(11)` としか記載されていない `datetime` 系カラムの実際の意味、キャンセル時の `fix` 解除ロジックの妥当性、祝日マスタの拠点共有範囲、営業時間ハードコード値の扱いは、要件ドキュメント4章で「未確認」とされていたため、本タスクで原典コードを直接確認した。

---

## 1. 原典コード調査結果(要件ドキュメント4章 未確認事項への回答)

### 1-1.(未確認事項4)`datetime` 系カラムの実際の型意味 → **UNIXタイムスタンプ(秒)であることを確認**

- `RsvController::regect()` (予約確定処理) 内:
  ```php
  $start = strtotime($start_date->format('Y-m-d H:i'));
  ...
  $new_dt = ['datetime' => $temptime, 'count' => 1, 'fix' => $fixflag, 'place' => $this->_place];
  ```
  → `rsv_datetime.datetime` は「日付+時刻(分単位)」を `strtotime()` でUNIX秒に変換した値。**日付のみのエンコード値ではなく、実時刻を含むタイムスタンプ**。
- `AdminController::holiday()` (祝日CSV取込) 内:
  ```php
  'datetime' => strtotime($line[0]),
  ```
  および `RsvController::judgepublicholidey()` 内:
  ```php
  $td_ux = strtotime($dt->format('Y-m-d'));
  $keyIndex = array_search($td_ux, array_column($this->_public_holiday, 'datetime'));
  ```
  → `rsv_holiday.datetime` は時刻を含まない「その日の0時」のUNIX秒。**日付のみを表す値として使われている**(時刻情報は常に00:00に固定)。
- `AdminController::add_fix()` (不定休登録) 内:
  ```php
  $fix->start = strtotime($target.' '.$start_time);
  $fix->end = strtotime($target.' '.$end_time);
  $fix->datetime = strtotime($target);
  ```
  → `rsv_fix.start`/`end` は日付+時刻を含むUNIX秒、`rsv_fix.datetime` は日付のみ(0時)のUNIX秒。ただし判定ロジック `RsvController::judgefix()` は **`start`/`end` のみを参照しており、`datetime` 列は判定に一切使われていない**(一覧表示等の補助情報に過ぎない)。

**結論**: 全て「UNIXタイムスタンプ(秒)」であり、`YmdHi` のような桁エンコード値ではない。ただし **タイムゾーンの扱いが本番環境で実際にどう設定されていたかは、公開リポジトリの `.env.default` からは断定できない**(後述「未決事項」参照)。

### 1-2.(未確認事項1)キャンセル時の `fix` 無条件解除バグ → **実装上のバグとして確認**

`AdminController::delete_rsv()`:
```php
foreach ($rsvs as $rsv) {
    $rsv->count--;
    if($rsv->fix) { $rsv->fix = false; }
    $this->RsvDatetime->save($rsv);
}
```
対象範囲内の各30分枠に対して、`count` を1減らした後の値が依然として `rsvlimit` 以上であるかを再判定せず、**`fix` が立っていれば無条件で `false` に戻している**。したがって、同一枠を複数の予約が使用しており、1件のキャンセルでは本来まだ満枠のままであるべきケースでも、満枠状態が解除されてしまう。**要件ドキュメントの懸念どおり、実装上のバグと断定できる**。

→ 新設計では `fix` という保存された真偽値そのものを廃止し、`count` と `BusinessHour.reservationLimit` の比較によって都度「満枠か」を判定する方式とした。これにより、このバグのクラス自体が構造的に発生しなくなる(詳細は3章)。

### 1-3.(未確認事項3)祝日設定の適用範囲 → **拠点非依存(全店舗共有)と確認。ただし「休診扱いにするか」は拠点別**

`RsvController::get_rsv()`:
```php
$this->_public_holiday = $this->RsvHoliday->find()->enableHydration(false)->toArray();
```
`place` によるフィルタが無く、`rsv_holiday` テーブル自体に `place` カラムも存在しない(`schema_ozdb.sql` で確認済み)。よって **祝日「日付」の登録は拠点を問わず共有される**。

一方、実際に祝日を休診扱いにするかどうかは、`RsvInite`(拠点別)の `week=7` 行の `open` フラグで決まる:
```php
$keyIndex = array_search(7, array_column($this->_config, 'week'));
$this->_public_holiday_confg = $this->_config[$keyIndex];
...
if(!$this->_public_holiday_confg['open'] && $keyIndex) { $rsv = false; }
```
→ **祝日マスタ(日付リスト)は共有、祝日を休診とするかの判断・祝日用の営業時間/予約上限は拠点別に設定可能**、という2層構造であることを確認した。この構造は業務要件として合理的(全国共通の祝日カレンダーを共有しつつ、店舗ごとの営業判断は独立)と考えられるため、新設計でもそのまま踏襲する。

なお、この判定コードには **`array_search()` の戻り値を素の真偽値として扱う実装バグ**も見つかった(該当祝日が配列の0番目要素にヒットした場合、`$keyIndex` が `0`(falsy)になり `!open && $keyIndex` が偽になるため、本来休診とすべき日が予約可能と誤判定される)。同様のバグは `RsvController::rsvcheck()` の枠占有チェックにも存在する。これは要件ドキュメント4章の「未確認事項2」に相当する懸念の実装上の裏付けであり、**新設計ではDBに対する存在確認クエリ(`findUnique`/`exists`相当)で判定するため、配列インデックスに依存するこの種のバグは構造的に発生しない**。

### 1-4.(未確認事項5)営業時間 9:00-18:30 ハードコードと `rsv_inite.opentime/closetime` の関係 → **9:00-18:30は絶対的な外枠(候補生成範囲)であることを確認**

`RsvController`:
```php
protected $_START_TIME = '9:00';
protected $_END_TIME = '18:30';
...
$this->_MAX_COUNT = (strtotime($this->_END_TIME) - strtotime($this->_START_TIME)) / (60 * $this->_STEP_MIN);
```
`get_rsv()` は `9:00` を起点に `_MAX_COUNT`(19コマ = 9:00〜18:30の30分刻み)だけループして候補枠を生成する。この外側ループの範囲は **全拠点・全曜日で固定であり、`rsv_inite.opentime/closetime` はこの範囲内でしか有効に機能しない**(`rsvcheck()` 内で `opentime`/`closetime` はこの候補枠の中で開閉判定に使われるのみ)。

→ **仮にある店舗が18:30以降や9:00以前の営業時間を設定しても、候補枠自体が生成されないため予約を受け付けられない**という制約が実装上存在することを確認した。これは業務要件として意図されたものか、単なる旧実装の技術的制約かは本タスクの調査範囲では判断できない。新設計のDBスキーマとしては `BusinessHour.openTime/closeTime` を拠点・曜日ごとに自由な時刻で保持できるようにし、**この9:00-18:30という外枠をDB制約としては一切設けない**こととした。この外枠を予約ロジック側で維持するかどうかは、DB設計の範囲外の業務判断として未決事項に記載する。

## 2. 全体設計方針

- 旧スキーマの `int(11)` エンコード列は廃止し、Postgresの `DateTime`(timestamptz)/`Date`/`Time` 型を用途に応じて使い分ける。
- 予約枠占有ロジックは「保存された真偽値(`fix`)」ではなく「実データ(`count`)と設定値(`reservationLimit`)の比較による都度判定」に置き換え、キャンセル時のバグを構造的に解消する。
- 拠点(日向/延岡)は `Place` マスタテーブルとして持たせる(理由は4章)。
- `rsv_users.address/sex/memo` は要件確定により引き継がない。
- 管理画面ログイン用に `AdminUser` を新設し、Auth.js Credentials Providerから参照する。
- 祝日・不定休・営業時間設定・予約枠・予約者の関連とそれぞれの拠点との対応は1-3節の調査結果に基づきモデリングする。

## 3. 採用案(モデルごとの設計判断)

### 3-1. `Place`(拠点マスタ)

現状「日向(1)」「延岡(2)」の2拠点固定だが、**enumではなくマスタテーブルとして持たせる**ことを採用した。

理由:
- Postgresのenumは値の削除や名称変更が難しく、将来拠点が増減した場合に単純な行追加/削除で対応できるテーブルの方が変更コストが低い。
- 各種テーブル(`BusinessHour`, `Closure`, `ReservationSlot`, `Reservation`)から`placeId`で参照する構造は、enumで直接値を持つ場合と比較して大きな複雑化にはならない(FK 1本の追加のみ)。
- MVPでは2拠点固定運用のままでよく、テーブル自体の追加コストは小さい(要件ドキュメント2章のMVP対象外「店舗を超えた拡張性」は今回のスキーマ設計とは独立に、後日拠点を追加する際もマイグレーション不要で対応できる)。

### 3-2. `AdminUser`(管理画面ログイン、旧: `users`)

- `password` → `passwordHash` に改名し、ハッシュ済み文字列のみを保持する方針を明記。
- `role` は旧実装の `inList(['admin','author'])` バリデーションをDBレベルのenum `AdminRole` として表現した。
- Auth.js (NextAuth v5) は Credentials Provider + JWTセッション運用を想定し、**NextAuthのDBアダプタが要求するテーブル群(`Account`/`Session`/`VerificationToken`)は設けない**。Credentials Providerの `authorize()` コールバック内で本テーブルを直接参照し、パスワード検証後はJWTにユーザー情報を積む方式を想定している。DBセッション管理(複数端末からの強制ログアウト等)が将来必要になった場合は別途アダプタ用テーブルの追加を検討する(未決事項参照)。

### 3-3. `BusinessHour`(曜日別営業設定、旧: `rsv_inite`)

- 旧 `week`(0-6 + 7=祝日)を `Weekday` enum(`SUNDAY`〜`SATURDAY` + `PUBLIC_HOLIDAY`)として明示化した。マジックナンバー `7` に依存する実装(旧コードの `array_search(7, ...)`)を排除する狙い。
- `opentime`/`closetime`/`breaktime_op`/`breaktime_cl` は旧実装では `"H:i"` 形式の可変長文字列だったが、時刻のみを表す値であるため Postgres の `TIME` 型に変更した。
- `rsvlimit` は旧schemaでNULL許容だったが、可用性判定の必須値であるため `reservationLimit Int @default(0)` としてNOT NULL化した(NULLのまま可用性計算に使われると実行時エラーやNaN類の不整合を招くリスクがあったため)。
- `@@unique([placeId, weekday])` を新設。旧実装には対応する制約が無く、「初期設定」機能(全削除→0〜7の8行再投入)が手続き的に重複を防いでいたに過ぎなかった。MVPでは「初期設定」機能自体はMVP対象外だが、通常の編集フローでも重複行が生まれないようDBレベルで保証する。

### 3-4. `PublicHoliday`(祝日マスタ、旧: `rsv_holiday`)

- 1-3節の調査結果のとおり、`placeId` を持たせず**全拠点共有**とした(旧実装の実態に合わせた)。
- `date` は日付のみを表す値であることが確認できたため `@db.Date` を採用し、`@unique` 制約を付与した(旧実装のCSV再投入運用に依存した重複防止ではなく、DBレベルで重複登録を防止する)。
- 「祝日を休診扱いにするか」「祝日の営業時間・上限」は `BusinessHour(weekday=PUBLIC_HOLIDAY)` 側で拠点別に管理する構造をそのまま踏襲した。

### 3-5. `Closure`(不定休、旧: `rsv_fix`)

- 旧実装の `start`/`end`(日付+時刻のUNIX秒)と `datetime`(日付のみ、判定には未使用)という冗長な3カラム構成を整理し、`date`(日付) + `isAllDay`(終日フラグ) + `startTime`/`endTime`(時刻のみ、終日でない場合のみ使用)の構成に変更した。
- 判定ロジック上必要なのは「日付」と「時刻範囲」であり、日付+時刻を毎回タイムスタンプとして持つ必要はないため、このように分解することで冗長性を排除した。
- `@@index([placeId, date])` を付与し、管理画面の不定休一覧・判定クエリの双方に対応する。

### 3-6. `ReservationSlot`(予約枠、旧: `rsv_datetime`)

最大の設計判断ポイント。旧実装は「実枠 (`count`)」と「派生的に決まるべき満枠フラグ (`fix`)」を**別々に、手動で同期する**構造になっており、これがキャンセル時のバグ(1-2節)の直接原因だった。

**採用案**: `fix` カラムを廃止し、`count`(実際の占有数)のみを保持する。可否判定は常に「`count >= BusinessHour.reservationLimit`」をアプリケーション層で都度計算する。

- `count` の増減は、予約確定時・キャンセル時ともに **`Reservation.startAt`/`endAt` から対象スロット群を都度再計算**して行う(30分刻みで機械的に再現できるため、対応関係を別テーブルで保持する必要がない)。これにより、予約確定時にincrementした枠と、キャンセル時にdecrementする枠が常に対称になり、旧実装のような「片方だけ処理される」不整合が起きにくい。
- `@@unique([placeId, startAt])` を新設。旧実装にはこの制約が無く、理論上は同時アクセス時に同一枠の重複行が作られ得た(`regect()` の「存在すればcount++、存在しなければ新規作成」という find-or-create 処理はトランザクション・ロックを伴っていない)。新設計ではこの制約とトランザクション内での `upsert` + 行ロックを組み合わせることで、レースコンディションに対して安全な形にする(具体的なロック戦略はAPI/ロジック設計フェーズで詳細化するが、方針は5章に記載)。
- `count` が負値にならないことは現状Prismaスキーマの機能だけでは表現できないため、マイグレーションSQLで `CHECK (count >= 0)` 制約を追加することを推奨する(未決事項参照)。

### 3-7. `Reservation`(予約者、旧: `rsv_users`)

- `address`/`sex`/`memo` は要件確定(3-3節)により引き継がない。
- 旧 `datetime_id`(占有スロットのうち最初の1件のみを指すFK)は廃止した。`startAt`/`endAt` があれば占有範囲全体を常に再現できるため、特定の1枠だけを指す中途半端なFKは不要と判断した。
- 旧 `time`(表示用文字列 `"9:00〜10:30"`)は廃止した。`startAt`/`endAt` から表示側で生成すればよく、DBに冗長な文字列を持たせる必要はない。
- 旧 `type`(フロントエンドが送信する自由文字列のラベル)は廃止した。`typeId` から一意に定まる表示ラベルであり、クライアント入力値をそのまま信頼して保存するのはデータ整合性上望ましくないため、ラベルはアプリケーション定数として `typeId` から導出する方針とした。管理画面の「種別」表示要件(要件ドキュメントE章)はこの導出方式で満たせる。
- `durationMinutes` を新設し、予約確定時点の所要時間をスナップショットとして保持する。`typeId ⇔ 所要時間` の対応が将来変更されても、過去予約の表示やスロット解放範囲の再計算が影響を受けないようにするため。
- `endAt` は `startAt + durationMinutes` から導出可能な冗長値だが、範囲検索(スロット占有判定、管理画面の日次一覧)を単純化する目的で非正規化して保持する。
- `@@index([placeId, startAt])` を付与し、管理画面の「対象日の予約一覧」表示・前日/翌日移動(要件E章)に対応する。

## 4. 代替案(検討したが不採用としたもの)

### 4-1. 拠点をenumにする案

`enum Place { HYUGA NOBEOKA }` として直接各テーブルにenum列を持たせる案も検討した。JOINが不要になりクエリがシンプルになる利点はあるが、将来拠点が増減する際にenumの変更(特に値の削除)はPostgresでは実質的に困難であり、マスタテーブル化のコスト(FK 1本)に対して見合わないと判断し不採用とした。

### 4-2. `ReservationSlotUsage` 中間テーブルによる完全な使用実績ベースの占有数管理

`ReservationSlot`に`count`を持たせず、`Reservation`と`ReservationSlot`の多対多中間テーブル(`ReservationSlotUsage`)を設け、占有数を都度 `COUNT(*)` で導出する案も検討した。これは「手動で増減するカウンタ」自体を完全に排除できるため理論上はさらに頑健(カウンタの増減漏れというバグクラスが原理的に存在しなくなる)だが、

- 中間テーブルが1つ増え、可用性計算のたびにJOIN集計が必要になる、
- 30分スロットと予約の対応が `startAt`/`endAt` から機械的に再計算できるため、中間テーブルによる明示的な記録は必須ではない、

という理由から、MVPでは過剰な複雑化と判断し不採用とした。ただし、将来的に「count不整合が疑われる」「監査ログとして誰がどの枠を使ったかを明示的に残したい」といった要求が出た場合は、この中間テーブル方式への移行を再検討する価値がある(5章のリスクにも関連)。

### 4-3. 予約のキャンセルを論理削除(ステータス列)にする案

旧実装は予約行を物理削除する方式であり、今回は要件の変更点として明示されていないため、`Reservation` も物理削除のまま踏襲することとした。監査要件(誰がいつキャンセルしたかの履歴を残す)が今後発生した場合は、`status` 列 + `cancelledAt` を追加する形での論理削除への切り替えを検討する。

## 5. リスク・技術的負債

- **同時実行制御はDBスキーマだけでは完結しない**。`ReservationSlot` への `count` 増減を安全に行うには、予約確定APIの実装で「対象スロット群を `upsert` → 行ロック(`SELECT ... FOR UPDATE` 相当、またはSERIALIZABLE分離レベル)→ 上限チェック → `count+1` とレコード挿入」を単一トランザクションにまとめる必要がある。本タスクではスキーマ設計のみを行っており、このトランザクション設計はAPI/ロジック設計フェーズで確定させる必要がある。
- **`count >= 0` のDB制約は未設定**。Prismaスキーマ言語だけでは表現できないため、マイグレーションSQLで `CHECK` 制約を追加することを推奨する。
- **旧本番データを移行する場合のタイムゾーン解釈**。6章「未決事項」参照。
- **祝日マスタが拠点共有であることの業務的な正しさ**は、実装事実としては確認できたが、それが「意図された仕様」か「たまたまplace列が無かっただけ」かは本タスクの調査範囲では判断できない。新スキーマは実装の実態を踏襲しているが、業務要件としての正式な確認が望ましい(要件ドキュメント4章の未確認事項3として引き続き残る)。
- **営業時間の9:00-18:30という外枠**をアプリケーションロジック側でどう扱うか(そのまま維持するか、`BusinessHour`の値を完全に信頼するかたちに変更するか)は、DBスキーマとしては両対応可能な形にしてあるが、業務判断が必要(要件ドキュメント4章の未確認事項5として引き続き残る)。

## 6. 未決事項

1. **本番DBの実際のタイムゾーン設定が未確認**。公開されている原典リポジトリの `config/.env.default` / `config/app.default.php` 上のデフォルトは `UTC` だが、実際の本番 `.env` が `Asia/Tokyo` 等に上書きされていたかは本リポジトリから確認できない。既存データを新DBへ移行する場合、`strtotime()` が実際にどのタイムゾームで解釈された値かを運用担当者に確認する必要がある(解釈を誤ると、移行後の予約時刻が数時間ずれる)。
2. **NextAuthのセッション管理方式**(JWTのみか、DBセッション/アダプタを併用するか)は認証設計フェーズで確定させる。DBアダプタを使う場合は `Account`/`Session`/`VerificationToken` 相当のテーブル追加が必要になる。
3. **`ReservationSlot.count` の非負制約**をマイグレーションSQLで追加するかどうか、また将来的に4-2節の中間テーブル方式へ移行する必要性があるかは、実装・運用状況を見ながら判断する。
4. **祝日マスタの拠点共有仕様、および営業時間9:00-18:30の外枠**は、要件ドキュメント4章の未確認事項3・5として、業務側への確認が引き続き必要(本タスクでは実装事実の確認のみ実施し、業務要件としての正当性の判断はスコープ外とした)。
