## 推奨する着手順の考え方(Product Owner 視点。ブランチ戦略は Architect に委ねる)

1. **土台(横断)**: US-003 を軸に予約中核ロジックの回帰テストを整備しつつリファクタ採用(architect.md 必須テスト対象の充足)。US-005 認証も同時に固める。
2. **利用者フロー確定**: US-001 → US-002 → US-003 → US-004。
3. **管理機能確定**: US-006/US-007/US-008/US-009(採用中心)+ US-010(新規実装で祝日ギャップ解消)。
4. **本番化**: US-017(設定実値化)→ US-015(インフラ)→ US-014(データ移行リハーサル)を経て初回リリース。
5. **リリース後の堅牢化**: US-011(非同期メール)、US-012(レート制限外部化)、US-016(監視)。
6. **要件確認後に判断**: US-013 / US-018 / US-019。
