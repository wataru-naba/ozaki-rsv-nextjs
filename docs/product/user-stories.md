# ユーザーストーリー(プロダクト制作フェーズ)

対象: Angular + CakePHP の予約システムを Next.js フルスタックへ再構築するプロジェクト。
基準: 検証・承認済み MVP(`mvp/feature/reservation-mvp`)。
参照: `docs/requirements/current-requirements.md`, `docs/design/db-schema.md`, `docs/design/api-design.md`, `docs/design/data-migration.md`, `.claude/agents/architect.md`

本ドキュメントはプロダクト制作フェーズ向けのユーザーストーリー(US)を定義する。
各 US は独立して実装・テスト・レビュー可能な単位とし、ID は Architect のブランチ命名規則
`feature/US-xxx-<name>` にそのまま使える形式(`US-001`〜)で付与する。
優先順位一覧は [backlog.md](./backlog.md) を参照。

---

## 利用者(患者)視点のストーリー

### US-001 相談内容(来店経験)の選択と所要時間の自動決定

**ユーザーストーリー**
As a 予約を取りたい患者,
I want to 拠点(日向/延岡)と自分の来店経験(はじめて/今月はじめて/今月来店済み)を選ぶ,
So that 自分に合った所要時間(90/60/30分)の枠で予約を進められる.

**ビジネス価値**
来店経験ごとに必要な施術・相談時間が異なるため、正しい所要時間で枠を確保することが
店舗運営の前提。ここを誤ると後続の空き状況・確定が全て破綻する起点となる。

**優先度**: Must

**受け入れ条件**
- Given 予約トップにアクセスした, When 拠点と来店経験を選ぶ, Then typeId(0/1/2)に応じた所要時間(90/60/30分)が確定し日時選択へ進める。
- Given 拠点または来店経験を変更した, When 次へ進む, Then 前に選んだ日時が破棄され、新しい条件で空き状況を取り直す。
- Given 未選択の状態, When 次へ進もうとする, Then 選択を促し先へ進めない。

**未確認事項**
- 来店経験の選択肢の表示文言(旧 entrance の文言をそのまま踏襲するか)はクライアント確認が望ましい。

**MVPコード再利用方針**: 2. リファクタリング後に採用(`app/reserve/steps/StepConsultation.tsx`)。

### US-002 空き状況カレンダーの閲覧(3段階表示・日付ページング)

**ユーザーストーリー**
As a 予約を取りたい患者,
I want to 選んだ拠点・所要時間に対する空き状況を、予約可能(○)/残りわずか(△)/予約不可の3段階で日付×時刻の表で見る,
So that 空いている日時を一目で把握して選べる.

**ビジネス価値**
空き状況の可視化は予約体験の中核。3段階表示は旧システムから継続する意図的な業務仕様であり、
残りわずか表示が予約の後押しになる。

**優先度**: Must

**受け入れ条件**
- Given 拠点と来店経験が確定している, When 日時選択画面を開く, Then 当日から21日先までの範囲で30分刻みの枠が3段階ステータスで表示される。
- Given 所要時間が90分の来店経験, When 空き状況を見る, Then 開始枠から所要時間ぶん連続する全30分枠が空いている場合のみ「予約可能/残りわずか」になる(1枠でも満枠・休憩・営業時間外なら不可)。
- Given ラストオーダー(日向5時間前/延岡1.5時間前)を過ぎた枠, When 空き状況を見る, Then その枠は予約不可として表示される。
- Given 祝日・終日休診・時間帯不定休に該当する枠, When 空き状況を見る, Then 予約不可として表示される。
- Given 残数が1〜3, When 空き状況を見る, Then 「残りわずか(△)」、残数4以上なら「予約可能(○)」と表示される。
- Given ページャーで日付を移動する, When 別の期間を表示する, Then 対応する期間の空き状況が取得・表示される。

**未確認事項**
- 「残りわずか」の残数の意味を旧「開始枠のみの数」から「区間内最小残数」へ変更している点(api-design.md 3.3/11章 未決事項5)の業務許容性は要確認。
- 空き状況範囲(当日〜21日先)を維持するか要確認(旧仕様どおり)。

**MVPコード再利用方針**: 2. リファクタリング後に採用(`lib/reservation/availability.ts`, `judge.ts`, `app/reserve/steps/StepDateTime.tsx`)。JST↔UTC の範囲取得の広め取り込みロジックの検証テストを伴う。

### US-003 予約の申込と確定(同時実行に安全な枠確保)

**ユーザーストーリー**
As a 予約を取りたい患者,
I want to 氏名・フリガナ・電話番号・メール・プライバシー同意を入力し、内容を確認して予約を確定する,
So that 希望日時の枠を確実に自分のものとして押さえられる.

**ビジネス価値**
予約の確定は本システムの最終目的。同時アクセス時の二重予約・満枠超過を防ぐデータ整合性は
店舗運営の信頼性そのもの。architect.md が必須テストを課す最重要領域。

**優先度**: Must

**受け入れ条件**
- Given 全必須項目を正しく入力した, When 確定する, Then 予約が1件作成され、占有する各30分枠の count が加算され、完了画面へ遷移する。
- Given フリガナがひらがな/カタカナ以外、電話番号が数字以外、メール形式不正、または未入力, When 確定する, Then 400 相当のバリデーションエラーとなり保存されない(氏名・カナも必須。旧フロントの必須抜けは踏襲しない)。
- Given 空き状況取得後に対象枠が満枠化・休診化した, When 確定する, Then サーバー側で全判定ステップを再検証し、確保できなければ 409 SLOT_UNAVAILABLE を返し、日時選択へ戻す。
- Given 同一枠に上限数まで同時申込が来た, When 複数リクエストを処理する, Then 上限を超える確定は発生せず、超過分は SLOT_UNAVAILABLE となる(同時実行テストで担保)。
- Given 90分予約で3枠を占有する, When 途中の枠だけ満枠, Then トランザクション全体がロールバックされ、部分的な枠確保が残らない。
- Given ハニーポットに値が入っている, When 送信する, Then 見かけ上成功を返しつつ実際には永続化されない。

**未確認事項**
- プライバシーポリシーの本文/リンク先はクライアント確認が必要。

**MVPコード再利用方針**: 2. リファクタリング後に採用(`lib/reservation/createReservation.ts`, `app/api/public/reservations/route.ts`, `app/reserve/**`)。同時実行・多枠ロールバック・再検証の回帰テストを実装物に含める。

### US-004 予約確認メールの受信(申込者宛 + 運用担当 Bcc)

**ユーザーストーリー**
As a 予約を確定した患者,
I want to 予約内容(店舗名・日時・氏名・カナ・電話・メール・両店舗連絡先・キャンセル案内)を記した確認メールを受け取る,
So that 予約が成立したことと詳細を手元で確認できる.

**ビジネス価値**
確認メールは申込者への必須の業務連絡であり、Bcc(運用担当)への通知は店舗側の受付把握に不可欠な
継続業務要件(確認済み)。メール到達は顧客安心と店舗オペレーションの両方に効く。

**優先度**: Must

**受け入れ条件**
- Given 予約が確定した, When 確定処理がコミットされた, Then 申込者宛に確認メールが送信され、設定された Bcc 宛にも同報される。
- Given メール送信が失敗した, When 予約は確定済み, Then 予約結果(201)には影響させず、失敗を構造化ログに残す(予約データは管理画面一覧が正)。
- Given メール本文, When 受信する, Then 要件C章の記載項目(店舗名・予約日時・氏名・カナ・電話・メール・両店舗電話番号・電話でのキャンセル案内)が含まれる。
- Given From/Bcc/件名, When 送信する, Then コードにハードコードせず環境変数から取得する。

**未確認事項**
- Bcc 送信先(現状 `k-katsuki@menicon.co.jp`)を本番でこの値のまま使うか要確認(US-017)。
- 送信元 From アドレス・件名の確定値は要確認。

**MVPコード再利用方針**: 1. そのまま採用可能(暫定)。本文・Bcc ロジックは採用。ただし信頼性は US-011 で非同期基盤へ置換する前提。

### US-005 管理画面へのログイン認証

**ユーザーストーリー**
As a 店舗スタッフ,
I want to ユーザー名とパスワードで管理画面にログインする,
So that 権限のある者だけが予約・設定を操作できる.

**ビジネス価値**
個人情報を含む予約データと営業設定を保護するセキュリティの前提。旧システムの
「公開/管理スコープ未分離」「実質認証なしの固定ID/PW」の欠陥を是正する。

**優先度**: Must

**受け入れ条件**
- Given 正しい資格情報, When ログインする, Then セッションが確立し `/admin/**` にアクセスできる。
- Given 誤った資格情報, When ログインする, Then 認証失敗としてエラーを表示し、セッションは確立しない。
- Given 未認証, When `/admin/**` にアクセスする, Then `/admin/login` へリダイレクトされる。
- Given 未認証で Server Action を直接呼び出す, When 実行する, Then 各 Action 先頭の認可チェックで拒否される(ページ保護だけに依存しない)。
- Given 公開ルート(`/`, `/reserve/**`, `/api/public/**`), When 未認証でアクセスする, Then 保護対象外として通常表示される。

**未確認事項**
- パスワードポリシー(最小長・ロックアウト等)を強化するか要確認。

**MVPコード再利用方針**: 1. そのまま採用可能(`lib/auth.ts`, `middleware.ts`, `lib/auth/session.ts`)。認証・認可の必須テスト追加のみ。

### US-007 予約詳細の確認とキャンセル

**ユーザーストーリー**
As a 店舗スタッフ,
I want to 予約の詳細を確認し、必要に応じてキャンセルする,
So that 患者からの電話キャンセル等に対応し、枠を再度開放できる.

**ビジネス価値**
キャンセルは日常的に発生する。旧実装はキャンセル時に満枠フラグを無条件解除するバグがあり、
他予約が使用中でも満枠が解除された。新設計はこのバグクラスを構造的に解消しており、その正しさの担保が価値。

**優先度**: Must

**受け入れ条件**
- Given 予約詳細画面, When キャンセルを実行する, Then 予約が削除され、占有していた各30分枠の count が startAt 昇順で -1 される。
- Given 同一枠を複数予約が使用中, When うち1件をキャンセルする, Then 他予約ぶんの占有(count)は正しく残り、満枠状態が誤って解除されない。
- Given count がすでに0の理論上の不整合枠, When キャンセルする, Then 処理は中断せず、警告ログを出して継続する(スタッフの目的達成を優先しつつ異常を可観測にする)。
- Given 存在しない予約ID, When キャンセルする, Then NOT_FOUND を返す。
- Given 未認証, When キャンセルを呼ぶ, Then UNAUTHORIZED で拒否される。

**未確認事項**
- キャンセル履歴(誰がいつ)を残す監査要件があるか要確認(現状は物理削除)。

**MVPコード再利用方針**: 2. リファクタリング後に採用(`app/admin/_actions/reservations.ts`)。データ整合性の必須テストを実装物に含める。

### US-008 曜日別営業設定(営業時間・休憩・予約上限)の編集

**未確認事項**
- 予約上限のデフォルト値・現実的な運用値は要確認。

### US-009 不定休の登録・削除

**ユーザーストーリー**
As a 店舗スタッフ,
I want to 拠点・日付単位で終日または時間帯の臨時休診を登録・削除する,
So that 通常営業日でも急な休診を予約受付に反映できる.

**ビジネス価値**
臨時休診は業務上頻繁に発生しうる。これを反映できないと誤って予約を受けてしまう。

**優先度**: Must

- Given 不定休を登録した, When その日時の空き状況を見る, Then 該当範囲が予約不可になる。
- Given 登録済みの不定休, When 削除する, Then 該当範囲の予約受付が再開される。
- Given 存在しない不定休ID, When 削除する, Then NOT_FOUND を返す。

**未確認事項**
- なし(設計・実装とも確定範囲)。

**MVPコード再利用方針**: 1. そのまま採用可能(`app/admin/_actions/settings.ts` の `createClosure`/`deleteClosure`, `app/admin/(dashboard)/slots/ClosureManager.tsx`)。

### US-010 祝日の個別追加・削除 + CSV 一括登録(MVP見送り分のギャップ対応)

**ユーザーストーリー**
As a 店舗スタッフ,
I want to 祝日マスタ(全拠点共有)に対して祝日を個別に追加・削除でき、さらに CSV(内閣府フォーマット)で一括登録できる,
So that 新設祝日や移動祝日を登録しつつ、年次の祝日をまとめて登録して祝日休診判定を最新に保てる.

**ビジネス価値**
祝日マスタを UI から更新できないと、運用開始後に新しい祝日を一切登録できず旧CSV運用より後退する。
これは単なる後回し可能な追加機能ではなく MVP の前提条件(api-design.md 5.5)。加えて、Product Owner の要望により
年次一括登録(旧 CakePHP の CSV 全削除→再投入と同一セマンティクス)を US-010 に統合し、
US-022(当初 Won't)を本 US へ取り込む(ADR 0002)。

**優先度**: Must

**受け入れ条件(個別追加・削除)**
- Given 未登録の日付, When 祝日として追加する, Then 全拠点共有の祝日マスタに1件追加される。
- Given 既に登録済みの日付, When 追加する, Then 重複として専用エラー(DUPLICATE_DATE)を返し、二重登録されない。
- Given 登録済みの祝日, When 削除する, Then マスタから除去される。
- Given 祝日を追加した, When 各拠点の祝日区分(PUBLIC_HOLIDAY)営業設定が休診, Then その日が予約不可になる(拠点別の休診判断は BusinessHour 側で決まる)。
- Given 祝日管理UI, When 表示する, Then 拠点非依存(全拠点共有)であることが明示され、拠点セレクタを設けない。

**受け入れ条件(CSV 一括登録=全削除→再投入。ADR 0002 / api-design.md 5.5.1)**
- Given 妥当な内閣府フォーマット CSV(ヘッダー行 + `YYYY/M/D,名称`、UTF-8/CRLF、0埋めなし日付), When アップロードする, Then ヘッダーがスキップされ日付が `YYYY-MM-DD` に正規化され、既存の祝日データが全削除された上で CSV の全行が再投入される。
- Given CSV に 1 件でも不正行(日付形式不正・実在しない日付・名称超過・ファイル内での日付重複)がある, When アップロードする, Then DB は一切変更されず(全削除も起きず)、不正行の行番号と理由が報告される(全か無か)。
- Given 全削除→再投入のトランザクション中に挿入が失敗する, When 処理する, Then 全削除ごとロールバックされ、既存の祝日データが保持される。
- Given 破壊的操作(全削除)である, When 実行しようとする, Then UI 上で「既存データを全削除して置き換える」旨が警告され、現在件数・取込予定件数を提示する確認を経て初めて実行される。
- Given 空ファイル・データ行 0 件・上限超過(1MB / 10,000 行)の CSV, When アップロードする, Then VALIDATION_ERROR となり、祝日が空になる事故や誤ファイル投入が防がれる。
- Given 未認証で一括登録アクションを直接呼ぶ, When 実行する, Then UNAUTHORIZED で拒否される。

**未確認事項**
- 本機能(個別 + CSV 一括)を正式に MVP/初回リリース範囲に含めることの最終承認(api-design.md 11章 未決事項1)。Product Owner としては「含める」を推奨。
- CSV 確認 UI の強度、文字コード自動判定(Shift_JIS)、手動追加祝日の保持要件、上限値の運用調整は実装・運用フェーズで確定(ADR 0002 未決事項)。

**MVPコード再利用方針**: 3. 再実装(個別追加・削除の UI・Server Action は `feature/US-010-holiday-management` で実装済み・未マージ。CSV 一括登録 Server Action・UI セクションを追加実装。設計 api-design.md 5.5 / 5.5.1、ADR 0002 に基づく)。

**必須テスト(architect.md データ整合性対象)**

- 全置換の原子性(createMany 失敗時に deleteMany がロールバックされ既存データ保持)。

### US-011 予約確認メールの非同期化・信頼性向上(Outbox + SQS + Lambda + SES)

**ユーザーストーリー**
As a 運用担当者,
I want to 予約確認メールを Outbox パターンで確実にキューへ積み、非同期(SQS + Lambda + SES)で冪等に配信する,
So that 一時的なメール障害でも通知が失われず、リクエスト処理と切り離して安定運用できる.

**ビジネス価値**
メール(申込者通知 + 運用 Bcc)は業務クリティカル。同期送信は障害時に取りこぼす。

**優先度**: Should

**受け入れ条件**
- Given 予約が確定した, When トランザクションがコミットされる, Then メール送信イベントが event_outbox に同一トランザクションで記録される。
- Given Outbox にイベントがある, When Publisher が処理する, Then SQS へ送出され、Lambda が SES で送信する。
- Given 同一イベントが再配信された, When Lambda が処理する, Then 冪等制御により二重送信されない。
- Given 送信が規定回数失敗した, When リトライ上限に達する, Then DLQ へ退避され、監視で検知できる(US-016 と連携)。
- Given 非同期化, When 予約確定 API を呼ぶ, Then メール処理はレスポンスをブロックしない。

**未確認事項**
- メール配信手段(SES 採用)の最終確定と送信ドメイン/SPF・DKIM 設定(クライアント/インフラ確認)。
- 初回リリースを同期方式(US-004)で開始し、本ストーリーを後続で適用する運用可否。

**MVPコード再利用方針**: 3. 再実装(`docs/aws-production-architecture.md` に基づく新規非同期基盤)。本文組立ロジック(US-004)は流用する。

2. 祝日個別管理(US-010)を初回リリース範囲に正式に含めることの承認。
3. 管理者ロール ADMIN/AUTHOR の権限差の要否と内容(US-013)。不要なら Won't 化。
4. 旧本番DBへの接続方法・移行の要否(US-014)。
5. 店舗の実電話番号・本番 From・Bcc 先・件名の確定値(US-004 / US-017)。
6. AWS アカウント・ドメイン・SES 送信ドメイン等の本番インフラ準備(US-015 / US-011)。
7. プライバシーポリシー本文/リンク、来店経験の選択肢文言(US-001 / US-003)。
8. キャンセル履歴(監査)を残す要件の有無(US-007)。
9. 利用者セルフキャンセル・残数表示など将来機能の要否(US-018 / US-019)。
