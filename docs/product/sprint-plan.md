# スプリント計画(プロダクト制作フェーズ)

対象: Angular + CakePHP の予約システムを Next.js フルスタックへ再構築するプロジェクト。
基準: 検証・承認済み MVP(`mvp/feature/reservation-mvp`)。
参照: [user-stories.md](./user-stories.md), [backlog.md](./backlog.md), `.claude/agents/architect.md`

本ドキュメントは、優先順位付き済みのユーザーストーリー(US-001〜US-022)を、
依存関係・リスク低減・クライアント確認待ちを踏まえて実行可能なスプリント単位へ割り付ける。

---

## 0. 前提と方針

### スプリント期間の仮定
- **1 スプリント = 2 週間** を仮定する(業界一般的な前提)。実際の期間・開始日はプロジェクト未確定のため、
  以下は **スプリント番号(Sprint 1〜7)ベース**で整理する。カレンダー日付は確定後に付与する。
- **チーム体制の仮定**: クロスファンクショナルな小規模チーム 1 つ(実装者複数 + レビュー)。
  1 スプリントあたり概ね 2〜4 ストーリー(採用中心の軽いものは並行実装可、再実装・インフラ系は重い)。

### 横断方針(全スプリント共通)
- **自動テストがプロジェクトに 1 件も存在しない**(user-stories.md 指摘)ため、Sprint 1 の最初にテスト基盤を立ち上げ、
  以降の各 US は **TDD**(受入条件→テスト先行→最小実装→リファクタ)で進める。受入条件に対応するテストは実装物の一部。
- architect.md が必須とするテスト対象(認証・認可・予約重複・データ整合性)は、該当 US のマージ条件に必ず含める。
- **ブランチ作成・分割・TDD 適用単位の最終判断は Architect の責任**。本計画は「どの US をどのスプリントで達成するか」
  (優先順位・依存関係・確認待ちの兼ね合い)に責任を持つ。特に US-015 は Architect による層別分割を前提とする。
- **Won't(US-020 / US-021 / US-022)は本計画の対象外**。

### スプリント数サマリ
| Sprint | ゴール要旨 | 含む US | 優先度構成 |
|---|---|---|---|
| 1 | テスト基盤・認証・利用者フロー前半 | US-005, US-001, US-002 | Must×3 |
| 2 | 予約確定・確認メール(利用者フロー完結) | US-003, US-004 | Must×2 |
| 3 | 管理機能(一覧・詳細/キャンセル・営業設定) | US-006, US-007, US-008, US-009 | Must×4 |
| 4 | 祝日管理・本番設定値反映(機能面のリリース準備) | US-010, US-017 | Must×2 |
| 5 | 本番インフラ構築・データ移行(初回リリース) | US-015, US-014 | Must×2 |
| 6 | リリース後の堅牢化 | US-011, US-012, US-016 | Should×3 |
| 7 | 要確認後に判断する将来機能(条件付き) | US-013, US-018, US-019 | Could×3 |

## 1. 依存関係の整理

### 依存グラフ(A → B は「B は A に依存」)

```
[認証土台]
  US-005 ─┬─→ US-006 ─→ US-007
          ├─→ US-008
          ├─→ US-009
          ├─→ US-010
          └─→ US-013

[利用者フロー]
  US-001 ─→ US-002 ─→ US-003 ─→ US-004
                │                  │
                │                  └─(本文組立を流用)→ US-011
                └─(空き状況へ設定が反映される)← US-008 / US-009 / US-010
              US-002 ─→ US-018
              US-007 ─(枠開放ロジック流用)→ US-019

[本番化基盤]
  US-004 ─(設定値の実値化)→ US-017
  US-015(インフラ)─┬─→ US-014(移行はステージング/新DB前提)
                    ├─→ US-011(SQS+Lambda+SES 前提)
                    ├─→ US-012(外部ストア選定と併走)
                    └─→ US-016(監視基盤と併走)
```

### 依存関係の要点
- **US-005(認証)は全管理機能(US-006/007/008/009/010/013)の前提**。最初に固める。
- **利用者フローは US-001 → US-002 → US-003 → US-004 の直列依存**。順序を崩せない。
- **US-002 の空き状況は US-008/009/010 の設定を反映する**が、判定ロジック自体は seed データでテスト可能なため、
  US-002 の着手は US-008/009/010 を待たない(ソフト依存)。エンドツーエンドの整合確認は US-010 完了後(Sprint 4)。
- **US-015(インフラ)は US-014・US-011・US-012・US-016 の共通前提**。ここがボトルネック。
- **US-014(移行)は US-015 のステージング/新 DB を前提にリハーサル**する。
- Could の 3 件はそれぞれ US-005 / US-002 / US-007 に依存しつつ、**クライアント確認結果で実施可否そのものが決まる**。

## 2. スプリント詳細

### Sprint 1 — テスト基盤・認証・利用者フロー前半

- **スプリントゴール**: 横断的なテスト基盤を確立し、認証と予約の入口(拠点/来店経験の選択 → 空き状況カレンダー)を固める。
- **含まれるストーリー**
  - US-005 管理画面ログイン認証(Must / 採用1)— 認証・認可の必須テスト追加。全管理機能の土台。
  - US-001 相談内容(来店経験)選択と所要時間の自動決定(Must / 採用2)。
  - US-002 空き状況カレンダー閲覧(3段階・日付ページング)(Must / 採用2)— JST↔UTC 範囲取得の検証テストを伴う。
  - (横断タスク)テストハーネス整備 — 各 US を TDD で進めるための前提。単一 US ではないが本スプリントで立ち上げる。
- **このスプリントを選んだ理由**
  - テスト不在が最重要ギャップのため、最初のスプリントで基盤を用意し以降の TDD を可能にする。
  - US-005 は他の全管理機能がブロックされないよう早期に固める必要がある。
  - 利用者フローは直列依存のため、起点(US-001)と可視化(US-002)を先行させ、Sprint 2 の予約確定に繋ぐ。
- **スプリント開始条件 / ブロッカー**
  - US-001 の来店経験の選択肢文言(確認7)、US-002 の残数算出方法変更の業務許容性(確認1)は**未解決でも進行可**。
    いずれも既存 MVP 実装の踏襲で暫定進行し、文言・業務許容の最終調整は表示層/しきい値で後追い吸収する。
    ただし確認1 が「不可」と判明した場合は US-002 の残数算出仕様を見直す軽微な手戻りが生じ得るため、早期確認を推奨。

### Sprint 2 — 予約確定・確認メール(利用者フロー完結)

- **スプリントゴール**: 同時実行に安全な予約確定と確認メール送信で、利用者フローをエンドツーエンドで完成させる。
- **含まれるストーリー**
  - US-003 予約の申込と確定(同時実行に安全な枠確保)(Must / 採用2)— 同時実行・多枠ロールバック・再検証の回帰テスト必須。
  - US-004 予約確認メール(申込者宛 + 運用 Bcc)(Must / 採用1・暫定)— 同期方式で暫定実装。
- **このスプリントを選んだ理由**
  - US-003 は US-001/US-002 完了後に着手可能。architect.md が必須テストを課す**最重要のデータ整合性領域**であり、
    集中して同時実行テストを整備する。
  - US-004 は US-003 のコミット後にメール送信が発火するため直後に置く。
- **スプリント開始条件 / ブロッカー**
  - US-003 のプライバシーポリシー本文/リンク(確認7)は**未解決でも進行可**(暫定コピー/リンクで実装し、本番文言は差し替え)。
  - US-004 の Bcc 先・From・件名(確認5)は**暫定の環境変数値で進行**し、**本番の確定値は US-017(Sprint 4)で反映**する。
    このスプリント内では確定値を必要としない。

### Sprint 3 — 管理機能(参照・キャンセル・営業設定)

- **スプリントゴール**: 店舗スタッフの日常運用機能(予約一覧・詳細/キャンセル・営業設定・不定休)を揃える。
- **含まれるストーリー**
  - US-006 予約一覧の閲覧・日付検索・前後日移動(Must / 採用1)。
  - US-007 予約詳細の確認とキャンセル(Must / 採用2)— データ整合性(count 減算・他予約非解除)の必須テスト。
  - US-008 曜日別営業設定の編集(Must / 採用1)。
  - US-009 不定休の登録・削除(Must / 採用1)— 未確認事項なし。
- **このスプリントを選んだ理由**
  - いずれも US-005(認証)に依存し、採用中心で並行実装しやすいため 1 スプリントにまとめる。
  - US-007 は旧実装のキャンセルバグを構造的に解消した正しさの担保が価値であり、テスト整備をここで行う。
  - US-008/US-009 の設定は US-002 の空き状況判定へ反映される(Sprint 4 の US-010 とあわせてエンドツーエンド整合を確認)。
- **スプリント開始条件 / ブロッカー**
  - US-007 のキャンセル履歴(監査)要件(確認8)は**未解決でも進行可**。現状の物理削除方式で実装し、
    監査要件が「必要」と判明した場合は履歴保存を追加対応(手戻りは限定的)。
  - US-008 の予約上限デフォルト値(軽微な確認)は seed の暫定値で進行可。

### Sprint 4 — 祝日管理・本番設定値反映(機能面のリリース準備)

- **スプリントゴール**: 祝日管理ギャップを解消し、本番の店舗連絡先・メール設定値を反映して、機能面でリリース可能な状態にする。
- **含まれるストーリー**
  - US-010 祝日の個別追加・削除(Must / 再実装3)— UI・Server Action 新規実装。祝日休診判定を最新に保つ MVP 前提条件。
  - US-017 店舗連絡先・メール送信設定の本番値反映(Must / 採用1)。
- **このスプリントを選んだ理由**
  - US-010 は US-005(認証)依存で、完了により US-002 の空き状況判定が祝日を含めて完全化する(利用者フローの整合完結)。
  - US-017 は US-004 完了後の設定値差し替えであり、本番リリース(Sprint 5)の直前に確定値を入れる位置が妥当。
- **スプリント開始条件 / ブロッカー(重要)**
  - **US-010 は「祝日個別管理を初回リリース範囲に含める承認」(確認2)がブロッカー**。Product Owner としては「含める」を推奨。
    承認前提で本スプリントに配置。**承認が得られない場合は本スプリントから外し**、代替運用(seed のみ)の可否を再判断する。
  - **US-017 は実電話番号・本番 From・Bcc 先・件名の確定値(確認5)が必須ブロッカー**。
    確定値が揃わないと**着手不可**。遅くとも Sprint 4 開始前までにクライアント確認を完了させる必要がある。

### Sprint 5 — 本番インフラ構築・データ移行(初回リリース)

- **スプリントゴール**: AWS 本番環境を構築し、旧本番データを移行して初回リリース可能な状態にする。
- **含まれるストーリー**
  - US-015 AWS 本番環境の構築とデプロイ(Must / 該当なし)— **Architect がインフラ層・CI/CD・シークレット管理等に分割する前提**。
  - US-014 旧本番データの移行(Must / 再実装3)— ステージングでリハーサル後に本番移行。
- **このスプリントを選んだ理由**
  - US-015 は他の全価値をユーザーへ届ける前提であり、リリース直前のこの位置に置く。規模が大きいため Architect が層別に分割。
  - US-014 は US-015 のステージング/新 DB を前提にリハーサルするため、US-015 と同一スプリントで連携させる。
  - 本スプリント完了をもって**初回リリース**とする(Sprint 1〜4 の Must 群 + インフラ + 移行)。
- **スプリント開始条件 / ブロッカー(最大のブロッカー群)**
  - **US-015 は AWS アカウント・ドメイン・SES 送信ドメイン等の本番インフラ準備(確認6)がブロッカー**。
    クライアント/インフラ確認が取れないと**着手不可**。準備はリードタイムが長いため、Sprint 1 と並行して早期に着手を要請する。
  - **US-014 は旧本番 DB への接続方法・移行の要否(確認4)がブロッカー**。接続方法(直接/エクスポート経由)が未確定だと
    スクリプトの前提が定まらず**着手不可**。実データの重複・不整合の有無も本番接続まで不明。

### Sprint 6 — リリース後の堅牢化(Should)

- **スプリントゴール**: メール非同期化・レート制限外部化・監視整備で、本番運用の信頼性を高める。
- **含まれるストーリー**
  - US-011 予約確認メールの非同期化・信頼性向上(Outbox + SQS + Lambda + SES)(Should / 再実装3)— US-004 の本文組立を流用。
  - US-012 レート制限の外部ストア化(複数インスタンス対応)(Should / 採用2)。
  - US-016 運用監視・アラート・DLQ・障害対応整備(Should / 該当なし)。
- **このスプリントを選んだ理由**
  - いずれも US-015(インフラ)を前提とし、初回リリース(Sprint 5)後に堅牢性を積み増す位置が妥当。
  - US-011 は US-004(同期・暫定)を非同期基盤へ置換。US-016 は US-011 の DLQ 監視を含むため US-011 と併走。
  - US-012 は複数インスタンス化時に制限が実質無効化する問題への対処で、本番規模での実効性を担保。
- **スプリント開始条件 / ブロッカー**
  - US-011 は SES 送信ドメイン/SPF・DKIM 設定(確認6、US-015 と同系)が前提。
  - US-012 は外部ストア(Upstash Redis / ElastiCache 等)選定が前提(US-015 のインフラ構成と併せて確定)。
  - US-016 は監視基盤(CloudWatch 等)とアラート通知先の確定が前提(US-015 と併せて)。

### Sprint 7 — 要確認後に判断する将来機能(条件付き)

- **スプリントゴール**: クライアント確認が取れた Could 機能を選択的に実装する(確認結果次第で内容が変動する条件付きスプリント)。
- **含まれるストーリー(いずれも確認結果に依存)**
  - US-013 管理者ロール別権限(ADMIN/AUTHOR)(Could / 採用2)— US-005 依存。
  - US-018 空き残数の数値表示(Could / 採用2)— US-002 依存。
  - US-019 利用者によるセルフキャンセル・変更(Could / 再実装3)— US-007 の枠開放ロジックを流用。
- **このスプリントを選んだ理由**
  - すべて**クライアント確認結果で実施可否そのものが決まる**ため、確認が揃うまで着手しない最後段に置く。
  - 確認が取れたものだけを本スプリントに投入する。取れないものは次以降へ持ち越すか、対象外化する。
- **スプリント開始条件 / ブロッカー**
  - **US-013 は ADMIN/AUTHOR の権限差の要否・内容(確認3)が未確定**。不要と判明した場合は **Won't 化**しスプリントから除外。
  - **US-018 / US-019 は将来機能の要否・方式(確認9、US-018 は残数表示の業務要否=確認1 とも関連)が未確定**。
    提供しないと判断された場合は着手しない。US-019 は本人確認方式の決定も必要。

### 最優先で先行取得すべき確認(リードタイム観点)
1. **確認6(AWS/ドメイン/SES 準備)** — 最大のリードタイム。**Sprint 1 と並行で即時着手を要請**。未準備は初回リリース全体を止める。
2. **確認4(旧 DB 接続方法)** — Sprint 5 の US-014 の前提。実データ調査にも時間を要するため早期確認。
3. **確認5(本番連絡先・メール値)** — Sprint 4 の US-017 の着手条件。Sprint 4 開始前までに確定必須。
4. **確認2(祝日管理の承認)** — Sprint 4 のスコープ確定に必要。

### 進行可能(暫定運用で吸収)な確認
- 確認1・7・8 は該当スプリントを暫定値/暫定コピー/物理削除で進行でき、後追い調整で吸収可能。
  ただし確認1 は US-002 の残数仕様に軽微な手戻りリスクがあるため早期確認が望ましい。

### 実施可否そのものが確認に依存(条件付きスプリント)
- 確認3・9 は Sprint 7(US-013 / US-018 / US-019)の実施可否と内容を左右する。確認が取れないストーリーは実施しない。

## 4. リリース区切りの整理

- **初回リリース = Sprint 5 完了時点**。Sprint 1〜4 の Must 機能群(利用者フロー + 管理機能 + 祝日 + 本番設定値)を、
  Sprint 5 の本番インフラ(US-015)へデプロイし、旧データ(US-014)を移行して稼働開始する。
