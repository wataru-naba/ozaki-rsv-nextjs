## 1. 要件一覧(機能カテゴリ別)

### A. 予約枠取得・カレンダー表示(利用者向け)
- 拠点(日向/延岡)ごとに予約枠一覧を取得するAPIがある。
- 取得範囲は当日から21日先まで、30分刻みで枠を生成する。
- 営業時間の基準は 9:00〜18:30。
- 判定順は「ラストオーダー制限 → 祝日判定 → 不定休・通常休診判定 → 予約枠使用状況判定」の4段階。
- ラストオーダー制限は拠点別(日向5時間前、延岡1.5時間前)で、直前予約を締め切る。
- 祝日は `rsv_holiday` マスタ、休診設定は `rsv_inite`(week=7=祝日扱い)を参照。
- 不定休(`rsv_fix`)は終日 or 時間帯単位で予約停止にできる。
- 予約種別(typeid)ごとに所要時間が異なり、開始枠から所要時間分の区間すべてが空いているかを判定する(複数枠にまたがる判定)。
- レスポンスの `rsv` 値は3段階の空き状況をフロントに伝える(**確認済み、詳細は3章参照**)。
- カレンダーUI上で「予約可能(○)/残りわずか(△)/予約不可」の3段階表示を行っている(`calender-form.component.html`, `yoyaku.css`)。

### B. 予約確定(申込)
- 利用者は氏名・フリガナ・電話番号・メールアドレス・来店経験(typeid)・希望日時を入力して送信する。
- typeid(来店経験)によって所要時間が変わる(**確認済み、詳細は3章参照**)。
- フロントバリデーション: フリガナは平仮名/カタカナのみ、電話番号は数字のみ、メールは形式チェック、プライバシー同意は必須。氏名系は `Validators.required` が未設定という実装上の抜けがある。
- サーバ側は所要時間ぶんの30分枠を走査し、既存枠があれば `count` を加算、`rsvlimit` に達したら `fix = true`(満枠)にする。区間内に既に `fix = true` の枠があれば登録を中断する。
- 予約者は `rsv_users` に1件、使用された各30分枠は `rsv_datetime` に反映される(1予約が複数枠を占有する設計)。
- `rsv_users` の `address`/`sex`/`memo` カラムは未使用(**確認済み、詳細は3章参照**)。
- 予約完了後、成功時のみ `http://www.ozaki-contact.jp/rsvcomplate/` にリダイレクトされる。

### C. 予約確認メール通知
- 予約確定成功時のみ、予約者宛に確認メールを送信する。
- From/Bcc/件名が固定。Bccは `k-katsuki@menicon.co.jp` にコードで直書きされている。**この通知は継続すべき業務要件(確認済み)**。
- 本文には店舗名・予約日時・氏名・カナ・電話番号・メール・両店舗電話番号・キャンセル時の電話連絡案内を含む。
- 未使用の `MailController`/`MailComponent` が別途存在するが、実運用には使われていない。

### D. 予約キャンセル(管理者操作)
- 利用者側にキャンセル機能はなく、管理画面からのみ操作可能。
- キャンセル時は「予約者削除」「使用中だった各30分枠の `count` を -1」「`fix = true` の枠を無条件で `false` に戻す」処理を行う。
- 実装上の懸念: 他の予約が同じ枠を使用中でも `fix` が無条件解除される可能性がある(**要確認、4章参照**)。

### E. 管理画面: 予約一覧
- 対象日の予約一覧表示、日付検索、前日/翌日移動。
- 表示項目: 予約時間、名前、種別、予約詳細、Email、TEL。
- キャンセル実行機能。

### F. 管理画面: 基本設定(営業設定)
- 曜日ごとの営業可否・診療時間・休憩時間・予約上限を `rsv_inite` で管理・編集する。
- 「初期設定」機能があり、対象店舗の `rsv_inite` を全削除して0〜7(日〜土+祝日)の8行を再投入する破壊的な初期化処理。

### G. 管理画面: 不定休診管理
- `rsv_fix` の一覧表示、終日休診/時間帯休診の追加・削除。

### H. 管理画面: 祝日取込
- `csv/syukujitsu.csv` を読み込み、`rsv_holiday` を全削除した上で再投入するCSV主導の再生成方式。

### I. 認証・拠点管理
- 予約系API(枠取得・確定)は固定のユーザー名/パスワードを送信しているが、**実際には認証処理として機能していない(確認済み、4章参照)**。
- 管理画面ログイン用に `users` テーブル(id/username/email/password/role)があり、`UsersController::login` による実質的なセッション認証が行われる。
- 拠点は「日向(place=1)」「延岡(place=2)」の2つがハードコードされている。

### J. ルーティング
- ルートURL `/` は `AdminHyuga::index`(管理画面トップ)に接続されており、公開API・利用者向け画面と管理画面のスコープ分離がされていない。

### K. データ保持(WordPress関連)
- 同一DB内にWordPress本体のテーブル群(`oc_*`)が存在するが、予約システムとの直接の関連は確認できない。

## 2. MVP範囲の提案

### MVP対象(初回リリースに必須)

| 機能 | 理由 |
|---|---|
| 予約枠取得・空き状況計算(拠点別、21日先まで、30分単位) | 予約システムの中核。 |
| 空き状況の3段階表示(予約可能/残りわずか/予約不可) | 実装として存在する意図的な業務仕様(確認済み)。 |
| 予約確定処理(複数枠占有・所要時間別判定・満枠判定) | 中核業務ロジック。 |
| ラストオーダー制限、祝日判定、不定休判定、通常休診判定 | 予約可否判定に不可欠な業務ルール。 |
| 予約確認メール通知(Bcc含む) | 申込者・運用担当者への必須の業務連絡(確認済み)。 |
| 管理画面: 予約一覧・キャンセル | 店舗スタッフの日常業務に必須。 |
| 管理画面: 基本設定(営業時間・休憩・予約上限)の閲覧・編集 | 営業日・時間帯変更時に必要。 |
| 不定休診の登録・削除 | 臨時休診は業務上頻繁に発生しうる。 |
| 拠点(日向/延岡)の切り替え | 現行運用の必須要件。 |
| 管理画面ログイン認証(適切な方式へ刷新) | セキュリティ上必須。予約枠取得・予約確定は認証不要な公開APIとして再設計してよい(確認済み)。 |

### MVP対象外(後回し候補)

| 機能 | 理由 |
|---|---|
| 祝日CSV一括取込・全削除再投入 | 旧デプロイ手法依存。UIでの個別追加編集で代替可能。 |
| `rsv_inite` の「初期設定(全削除→0-7再投入)」 | 通常運用フローには不要。破壊的操作のため優先度低。 |
| 未使用の `MailController`/`MailComponent` | 現行でも未使用のデッドコード。 |
| WordPress関連テーブル・機能(`oc_*`) | 予約システムとの関連未確認、別途スコープ判断が必要。 |
| サンプル画面系のtypeid(1〜3系統) | 実運用は entrance の 0〜2 系統のみ。 |
| 店舗を超えた拡張性(3店舗目以降) | 現行はハードコードされた2拠点構成。要件として未明言。 |

## 3. 確認済み事項(原典コード調査・ユーザー回答により解消)

### 3-1. `param.rsv` の意味と表示仕様
- `false` = 予約不可 / `1〜3` = 予約可能・空き少なめ / `4以上` = 予約可能 / `true` = 予約可能(枠数を持たないケース)。
- 根拠: `RsvController.php` の `rsvcheck()`。
- フロント(`calender-form.component.html`)で `rsv > 3` → 予約可能(○/赤)、`1〜3` → 残りわずか(△/青)、`false` → 予約不可、と実際に3段階表示を行っている。**意図的な業務仕様**。

### 3-2. typeid(来店経験)と所要時間の対応
| typeid | 来店経験 | 所要時間 |
|---|---|---|
| 0 | 一度も来店したことがない・わからない | 90分 |
| 1 | 今月ははじめて来店する | 60分 |
| 2 | 今月に来店した事がある | 30分 |
- 根拠: `rsv.service.ts:95`(typeidをそのまま予約APIに送信)、`entrance.component.ts:33`(選択肢定義)。
- api-list.mdに記載の「サンプル画面系typeid 1〜3(新患/準患/再来)」は別体系で実運用外。

### 3-3. `rsv_users` の address/sex/memo カラム
- 未使用カラムと確定。`RsvUsersTable.php` のバリデーションはコメントアウト済み、保存処理・フロント入力フォームのいずれにも登場しない。
- **Next.js版DB設計に引き継ぐ必要なし**。

### 3-4. 固定認証情報(admin/yash7208 等)と管理画面ログインの関係
- フロントから送信されるID/PW(予約枠取得は`admin/yash7208`、予約確定は`admin/admin`とパスワードが異なる)は**不要な情報であり、実際には認証を行っていない**(ユーザー確定回答)。
- 管理画面ログイン(`UsersController::login`, `users`テーブル)のみが実質的な認証機能。
- **Next.js移行方針**: 予約枠取得・予約確定エンドポイントは最初から「認証不要な公開API」として設計し直す(必要ならCSRF対策・レート制限で保護)。固定ID/PWの送信・検証は廃止してよい。

### 3-5. メールBcc(`k-katsuki@menicon.co.jp`)
- **継続すべき業務要件**(ユーザー確定回答)。旧システムでは `RsvController.php` の `sedding()` にハードコード。
- **Next.js移行方針**: Bcc送信自体は維持する。ハードコードのままか環境変数化するかは実装時に判断(環境変数化が望ましい)。

## 4. 残る未確認事項・要確認ポイント

1. **キャンセル時の `fix` 解除ロジックの妥当性**: キャンセル時に該当枠の `fix` を無条件で `false` に戻しており、他の予約がまだ枠を使用中でも満枠状態が解除される可能性がある。業務要件として許容される仕様か、修正すべきバグかの判断が必要。
2. **`array_search()` の0番目ヒット誤判定**(祝日判定・開始枠検索の実装上の懸念): 実際に業務影響(誤って予約不可/可となるケース)が発生しているか未確認。
3. **祝日設定の適用範囲**: 祝日マスタ(`rsv_holiday`)が拠点間で共有される(拠点別に個別設定できない)仕様が業務要件として正しいか。
4. **`rsv_fix`/`rsv_datetime`/`rsv_holiday` の `datetime` カラムの実際の型意味**: DBスキーマ上は `int` とだけ記載されており、UNIX時刻か日付のみを表す値か等、正確なデータ形式が未確認。
5. **営業時間 9:00-18:30 のハードコード値と `rsv_inite.opentime/closetime` の関係**: コード上の外枠(候補生成範囲)なのか、店舗側で設定可能な時間帯を上書きできない絶対的な上限なのか未確認。仮に18:30以降の営業に対応できない制約であれば業務要件として問題になりうる。

### 旧システム都合と推測される要素(移行不要の可能性が高いもの)
- 祝日マスタのCSV全削除・再投入という運用(旧デプロイ手法に依存)。
- ルートURL `/` が管理画面トップに接続されている構成(公開/管理のスコープ未分離)。
- `RsvUsersTable` のバリデーションがコメントアウトされている点(技術的負債であり業務要件ではない)。
- 未使用の `MailController`/`MailComponent` の残存。
- URLパスの大文字小文字ゆれ(`rsvHyuga` のみ小文字)。
