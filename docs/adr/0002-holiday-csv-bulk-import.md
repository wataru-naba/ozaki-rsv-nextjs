# ADR 0002: 祝日の CSV 一括登録(全削除→再投入)を US-010 に追加する

- Status: Accepted
- Date: 2026-07-22
- Deciders: Architect
- 関連: `docs/product/user-stories.md`(US-010 / US-022), `docs/design/api-design.md`(5.5 節), `docs/design/db-schema.md`(3-4 `PublicHoliday`), `docs/requirements/current-requirements.md`(H 章), ADR 0001
- 対象 US: US-010(`feature/US-010-holiday-management`、個別追加・削除は実装済み・未マージ)

---

## 背景

- Product Owner から、US-010(祝日の個別追加・削除)に対し **CSV による祝日の一括登録** を追加する要望が出た。挙動として「CSV 一括登録時は旧データを全削除して置き換える(全削除→再投入)」ことが明示されている。
- この機能はもともと **US-022「祝日の CSV 一括取込」= Won't(現フェーズ対象外)** に分類されていた。理由は「旧デプロイ依存。US-010 の個別 UI 編集で代替」。今回の要望はこの判断を意図的に覆し、CSV 一括取込を US-010 のスコープへ取り込むもの。
- 旧 CakePHP 実装(`AdminController::holiday()`)は、内閣府提供の `csv/syukujitsu.csv` を読み込み `rsv_holiday` を **全削除した上で再投入** する方式だった(要件ドキュメント H 章)。今回の要望はこの旧セマンティクスと同一。
- Product Owner から提供された実サンプル `syukujitsu.csv` は内閣府「国民の祝日・休日」CSV と同一フォーマット。
  - ヘッダー行: `国民の祝日・休日月日,国民の祝日・休日名称`
  - データ行例: `1955/1/1,元日`(日付は `YYYY/M/D`、0 埋めなし)
  - 文字コード UTF-8、改行 CRLF、全 1067 行(1955〜2027、約 26KB)。
- 既存 DB 設計(`db-schema.md` 3-4)では `PublicHoliday` は `date`(`@db.Date` かつ `@unique`)と `name`(任意)のみ。拠点非依存(全拠点共有)。US-010 の個別追加・削除は Server Action(`createPublicHoliday` / `deletePublicHoliday`)として実装済み。

---

## 要件

1. `/admin/holidays` 画面から CSV(内閣府フォーマット)をアップロードして祝日を一括登録できる。
2. 一括登録は **既存の祝日データを全削除してから CSV の内容で再投入** する(全削除→再投入)。旧仕様と同一セマンティクス。
3. 破壊的操作(全削除)であることを利用者に明示的に警告し、誤操作を防ぐ。
4. 既存の個別追加・削除機能(US-010 実装済み)の設計を壊さない。
5. 不正な CSV(パース不能・日付形式不正)で既存データを破壊しない。データ整合性を最優先する(architect.md 必須テスト対象)。
6. 誤ファイル(巨大ファイル・無関係 CSV)アップロードへの防御を持つ。

---

## 設計方針

- **既存の US-010 実装パターン(Server Action + `ActionResult` + `requireAdminSession` + `revalidatePath`)に一貫して乗せる**。CSV 取込のためだけに `/api/admin/**` を新設しない(api-design.md 6 章「管理更新系は Server Action、`/api/admin` は原則設けない」を踏襲)。
- **破壊(全削除)の前に完全検証(全行バリデーション)を終える**。1 行でも不正があれば **DB に一切触れずに中断** し、行番号付きでエラーを報告する。全削除→再投入という破壊性ゆえ、部分成功(不正行をスキップして投入)は「祝日が静かに欠落する」危険があるため採らない。
- **全削除と全件挿入を単一トランザクションで実行** し、途中失敗時は削除ごとロールバックさせる。
- 拠点非依存(全拠点共有)である点は個別管理と同一。拠点セレクタは設けない。

---

## 採用案

### 1. 動線・UI: 既存 `/admin/holidays` に CSV 一括登録セクションを追加

- 別画面・別動線は設けず、祝日管理画面(`HolidayManager`)内に「CSV 一括登録」セクションを追加する。祝日管理の自然な置き場所であり、個別追加・削除と同一画面で完結する。
- セクションは個別追加フォームと視覚的に明確に区切り、**「この操作は既存の祝日データをすべて削除し、CSV の内容で置き換えます」** という破壊的操作の警告を常時表示する。
- 実行前に **確認ダイアログ**(現状の削除ボタンと同様の `window.confirm` 相当、望ましくはモーダル)で「既存 ◯ 件を削除し、CSV の △ 件で置き換えます。よろしいですか?」と現在件数・取込予定件数を提示して同意を取る。
- 結果表示: 成功時は取込件数、失敗時は「エラー行番号 + 理由」を一覧表示する。

### 2. 処理方式: Server Action(FormData で File を受け取る)

- Next.js App Router の Server Action は `FormData` 経由で `File` を直接受け取れるため、Route Handler を新設せず Server Action `importPublicHolidaysCsv(formData: FormData)` を採用する。
- 採用理由:
  - 既存 US-010 の 2 アクションと同じ層・同じ認可(`requireAdminSession`)・同じ `revalidatePath` に揃えられ、実装/レビューが一貫する。
  - `/api/admin/**` を新設すると別途認証・CSRF 対策が必要になり、api-design.md 6 章の方針にも反する。
  - CSV は約 26KB と小さく、Server Action のボディサイズ既定(1MB)に十分収まる。
- 戻り値は既存の `ActionResult<T>` に合わせる。成功: `{ ok: true, data: { importedCount } }`。失敗: `ActionResult` の `error`(下記コード)。

### 3. CSV パース方針

- **読み込み**: `File.text()` で UTF-8 文字列として取得。先頭 BOM があれば除去。改行は CRLF/LF 双方を許容。
- **ヘッダー行**: 先頭行が既知ヘッダー(`国民の祝日・休日月日,国民の祝日・休日名称`)または「1 列目が日付として解釈できない行」の場合はヘッダーとしてスキップする。末尾の空行は無視する。
- **日付**: `YYYY/M/D`(0 埋めなし)を正規表現 `^\d{4}/\d{1,2}/\d{1,2}$` で受理し、実在日(例 `2026/2/30` は不正)を検証した上で `YYYY-MM-DD`(0 埋め)へ正規化する。
- **名称**: 2 列目を `name` に格納(50 文字上限。既存 `PublicHoliday.name` の設計に合わせ超過は検証エラー)。空は `null`。
- **不正行の扱い(全か無か)**: **全行を先に検証し、1 件でも不正(日付形式不正・実在しない日付・名称超過・ファイル内での日付重複)があれば、DB を一切変更せずに中断** し、`VALIDATION_ERROR` として不正行の「行番号 + 理由」をまとめて返す。全行が妥当な場合のみトランザクションへ進む。破壊的操作の前提として部分投入・サイレントスキップは禁止する。
- **ファイル内重複日付**: `PublicHoliday.date` は `@unique`。ファイル内に同一日付が複数あると `createMany` が制約違反になるため、検証段階でファイル内重複を検出し `VALIDATION_ERROR` として報告する(誤ファイルの兆候として中断)。

### 4. トランザクション設計(全削除→再投入)

- 配列形式の単一トランザクションで実行する:
  ```ts
  await prisma.$transaction([
    prisma.publicHoliday.deleteMany({}),           // 既存を全削除
    prisma.publicHoliday.createMany({ data: rows }) // 検証済み行を一括挿入
  ]);
  ```
- 両操作が 1 つの DB トランザクションで走るため、`createMany` が失敗すれば `deleteMany` もロールバックされ、**既存データは保持される**(整合性保証)。
- 事前に全行検証済み(かつファイル内重複排除済み)のため、通常 `createMany` は `@unique` 違反を起こさない。`skipDuplicates` は使わない(サイレントスキップを避け、想定外の重複は失敗として顕在化させる)。
- 成功後に `revalidatePath("/admin/holidays")`。
- **個別追加・削除との整合性/競合**: 本操作は同一 `PublicHoliday` テーブルに対する全置換。想定運用は単一管理者・小規模のため、一括取込中に別セッションが個別追加する競合の実害は極めて低い。DB トランザクションが原子性を保証するため、競合しても最終状態は「個別操作 or 一括操作のどちらかが勝つ」一貫状態になり、部分破損は起きない。一括登録は排他的な破壊操作として扱う旨をドキュメント・UI に明記する(厳密な排他制御=アドバイザリロック等は MVP では過剰とし未導入)。

### 5. 誤アップロード防御(上限)

- ファイルサイズ上限: 1 MB(実サンプル約 26KB に対し十分な余裕。無関係な巨大 CSV を弾く)。
- 行数上限: 10,000 行(73 年分 1067 行に対し余裕。桁違いの誤ファイルを弾く)。
- 空ファイル・データ行 0 件は `VALIDATION_ERROR`(全削除だけが実行されて祝日が空になる事故を防ぐ)。
- 拡張子/Content-Type は `.csv` / `text/csv` を目安に補助的チェック(厳格拒否はせず、本質はパース検証で担保)。

### 6. エラーコード

- 既存 `ActionErrorCode` を再利用する(`VALIDATION_ERROR` / `UNAUTHORIZED` / `INTERNAL_ERROR`)。CSV 固有の詳細(不正行の行番号・理由)は `fieldErrors` 相当の構造 or `message` に集約して返す。新規コードは原則追加しない(個別追加の `DUPLICATE_DATE` は本操作では発生しない=全削除後に投入するため)。

## 代替案

- **Route Handler(`POST /api/admin/holidays/import`)方式**: multipart を素直に扱えるが、認証・CSRF・エラー整形を独自実装する必要があり、api-design.md 6 章の「管理更新系は Server Action」方針に反する。既存 2 アクションとの一貫性を優先し不採用。
- **差分マージ(upsert)方式**: 既存を残し CSV 分だけ追加/更新する非破壊方式。データ消失リスクは下がるが、Product Owner 要望(旧仕様どおりの全削除→再投入)と異なり、旧マスタに残った手動追加祝日の去就が曖昧になる。要望に忠実に全置換を採用。将来 CSV に無い手動祝日を保持したい要件が出たら再検討(未決事項)。
- **不正行スキップ+レポート方式**: 一部エラーを飛ばして投入。UX は寛容だが、全削除を伴うため「祝日が静かに欠落」する重大リスク。破壊操作では全か無かが安全で不採用。
- **クライアント側パース**: ブラウザで解析して整形済み JSON を送る案。実装は軽くなるが、検証を信頼境界の外(クライアント)に置くことになり整合性上不可。サーバー側で検証する。
- **ドライラン(検証専用)API の分離**: 取込前に別アクションでプレビュー件数を返す 2 フェーズ方式。UX 上は理想だが MVP には過剰。単一アクションが「破壊前に全検証する」設計で安全性は担保できるため、プレビューは将来拡張とする。

## リスク

- **文字コード依存**: 本サンプルは UTF-8 だが、内閣府 CSV は歴史的に Shift_JIS 配布だった時期がある。Shift_JIS ファイルを UTF-8 として読むと名称が文字化けする。MVP は UTF-8 前提とし、将来の文字コード自動判定を拡張候補として残す(未決事項)。
- **全削除の破壊性**: 確認ダイアログと事前全検証で二重に防御するが、正しいが古い CSV を投入すれば新設祝日が消えうる。運用手順として「最新の内閣府 CSV を使う」ことを利用者ガイドに明記する。
- **必須テスト**: 本機能は「データ整合性」に該当(architect.md 必須テスト対象)。少なくとも「不正行検出時に DB 未変更(全削除も起きない)」「全置換の原子性(createMany 失敗時ロールバック)」「ヘッダー/CRLF/BOM/0 埋めなし日付の正規化」「サイズ・行数上限」をテストで担保する(実装フェーズの必須要件)。
- **競合**: 一括取込中の個別追加は原子性で最終一貫だが「消えたように見える」体験がありうる。単一管理者運用前提で許容し、UI 文言で一括登録の全置換性を明示する。

## 未決事項

1. **確認 UI の強度**: `window.confirm` で足りるか、件数表示付きモーダル+チェックボックス同意まで求めるかは実装フェーズで UX を確定(本 ADR は「破壊性の明示 + 事前全検証」を必須要件として確定)。
2. **文字コード自動判定(Shift_JIS 対応)**の要否。MVP は UTF-8 前提。
3. **手動追加祝日の保持要件**: 将来 CSV に無い手動祝日を残したい要件が出た場合、全置換ではなく差分マージへ切り替えるか(現状は Product Owner 要望どおり全置換)。
4. **ファイルサイズ/行数の具体上限値**(1MB / 10,000 行)は目安であり運用実績で調整可。
