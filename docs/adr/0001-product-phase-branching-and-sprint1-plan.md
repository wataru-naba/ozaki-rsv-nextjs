# ADR 0001: プロダクト制作フェーズのブランチ方式と Sprint 1 実行計画

- Status: Accepted
- Date: 2026-07-16
- Deciders: Architect
- 関連: `.claude/agents/architect.md`(ブランチ運用責任 / TDD / マージ条件), `docs/product/sprint-plan.md`, `docs/product/user-stories.md`, `docs/design/api-design.md`
- 対象スプリント: Sprint 1(US-005, US-001, US-002)

---

## 背景

- 対象は Angular + CakePHP の予約システムを Next.js フルスタックへ再構築するプロジェクト。
- MVP は検証・承認済み(`mvp/feature/reservation-mvp`)。Product Owner がユーザーストーリー・バックログ・スプリント計画を作成済み。
- architect.md の方針により「MVP 検証完了後、MVP ブランチを無条件で `main` にマージしない」を守った結果、
  **`main` には実際の Next.js アプリコードが存在しない**。現在 `main` にあるのは `docs/`, `.claude/`,
  `prisma/schema.prisma`, `schema_ozdb.sql` のみ。動作するアプリ(`app/`, `lib/`, `package.json`,
  `prisma/migrations/` 等、約 74 ファイル)は `mvp/feature/reservation-mvp` にのみ存在する。
- この状態のまま `feature/US-xxx-<name>` を `main` から切っても、Next.js の土台(ビルド設定・Prisma・テスト基盤)
  が無く着手できない。一方で MVP ブランチをそのまま `main` にマージすると、architect.md の
  「MVP コードを 1.そのまま採用 / 2.リファクタ後採用 / 3.再実装 / 4.廃棄 に分類する」原則を回避してしまう。
- さらに **自動テストがプロジェクトに 1 件も存在しない**(user-stories.md 指摘)。プロダクト化は TDD 前提のため、
  テスト基盤の立ち上げ位置も同時に決める必要がある。

---

## 要件

本 ADR が満たすべき判断基準(タスク指示より):

1. architect.md の「MVP コードをファイル単位で 4 分類する」方針を**技術的に実現できる**こと。
2. Sprint 1(US-005 / US-001 / US-002)を**今すぐ着手可能**にする現実性。
3. 将来スプリントでも**繰り返し使える再現性**のある方法であること。
4. マージ条件(受入条件充足・必須テスト成功・静的解析成功・重大な脆弱性なし・MVP 専用仮実装が残っていない・
   秘密情報/デバッグコードなし)を各 US で担保できること。

---

## 設計方針

### 検討した選択肢

- **案A(土台先行 PR + 各 US は `main` 起点)**: 「そのまま採用可能」なプロジェクト基盤だけを土台 PR で `main` に入れ、
  各 US を `main` から切る。
- **案B(各 US を `mvp` ブランチ起点)**: 各 US を `mvp/feature/reservation-mvp` から分岐し、TDD で改修して `main` へマージ。
- **採用: 案A の土台先行 + ファイル単位ポート(ハイブリッド)**。

### 案B を採らない理由

- `mvp` から分岐すると、その US に無関係な約 74 ファイル(3.再実装 / 4.廃棄 候補を含む)を丸ごと引き込む。
  US ブランチを `main` にマージした時点で、**未分類・未レビューの MVP コードが `main` に流入**し、
  「無条件でマージしない」「ファイル単位で 4 分類する」という核心原則を骨抜きにする。
- PR が巨大化・無関係差分だらけになり、レビュー不能。マージ条件(仮実装が残っていない等)の検証が困難。
- インメモリ・レート制限、同期メール、店舗電話番号のプレースホルダ等 **MVP 専用仮実装**が、
  担当外 US 経由で `main` に紛れ込むリスクが高い。

### 案A(単独)の限界と、その補完

- 案A 単独では「土台=そのまま採用可能な基盤」の境界が曖昧になりがち。**土台にはフィーチャーコードを含めない**と厳密に定義する。
- 各 US が「MVP コードを起点にする」手段として、**`mvp` ブランチからファイル単位で取り込む**方式を組み合わせる:

  ```
  git switch -c feature/US-xxx-<name> main
  git checkout mvp/feature/reservation-mvp -- <その US に該当するファイルパスのみ>
  # → 取り込んだファイルを TDD で改修(2.は必ずリファクタ、3.は新規実装)し、必須テストを追加してからコミット
  ```

  これにより、**`main` に入るのは意図的に選択・レビューされたファイルだけ**になり、
  architect.md の 4 分類がコミット単位でそのまま表現される(再現性のある標準手順)。

---

## 採用案

### 決定 1: 土台 PR(`chore/adopt-mvp-scaffold`)を先に `main` へ入れる

`main` から `chore/adopt-mvp-scaffold` を切り、**カテゴリ 1(そのまま採用可能)の純粋なプロジェクト基盤のみ**を
`mvp` からポートして `main` へマージする。**フィーチャー(業務ロジック・画面)は一切含めない**。

土台 PR に含めるもの(build / tooling / DB / test harness / app shell):

| 区分 | ファイル(例) | 目的 |
|---|---|---|
| ビルド/依存 | `package.json`, ロックファイル, `.nvmrc`, `.gitignore` | Next.js が動く土台 |
| TS/Next 設定 | `tsconfig.json`, `next.config.*`, `next-env.d.ts` | ビルド・型 |
| 静的解析 | ESLint / Prettier 設定 | マージ条件「静的解析成功」の基準を定義 |
| スタイル土台 | Tailwind/PostCSS 設定, `app/globals.css`(最小) | 各画面の共通前提 |
| App シェル | `app/layout.tsx`(ルート最小), 必要最小の `app/page.tsx` プレースホルダ | ビルド可能な骨格。**トップ画面本体は US-001 で実装** |
| DB クライアント | `lib/prisma.ts`(シングルトン) | 全 US 共通の DB 接続基盤 |
| Prisma | `prisma/schema.prisma`(main と MVP を突合し確定), `prisma/migrations/**`, `prisma/seed.ts` | 実行可能な DB。**schema 突合は本 PR のタスク** |
| 環境変数 | `.env.example` | 秘密情報を持たず変数名だけ提供 |
| テスト基盤 | Vitest 設定(`vitest.config.ts`)+ セットアップ + スモークテスト 1 本 | **全 US の TDD 前提。ここで一括立ち上げる** |
| npm scripts | `test`, `lint`, `typecheck`, `build` | マージ条件で参照する検証コマンド |

**土台 PR に含めないもの(=各 US が持ち込む)**: `middleware.ts` と `lib/auth.*`(US-005)、`app/reserve/**`(US-001/002)、
`lib/reservation/**`(US-002 以降)、`lib/mail/**`・`lib/api/rateLimit.ts`・各 Server Action(後続スプリント)。

**土台 PR のテスト方針**: 土台は認証・認可・金額・在庫・予約重複・データ整合性のいずれにも触れないため、
architect.md の必須テスト対象は発生しない。テストハーネスが動くことを示す**スモークテスト 1 本**(例: 純粋関数 or Prisma クライアント import)で足りる。フィーチャーのテストは各 US で書く。

### 決定 2: テスト基盤は US-005 ではなく土台 PR で立ち上げる

Vitest 設定は全 US 共通のインフラであり、特定フィーチャー(US-005)に結合させると US-001/US-002 が待たされるか重複導入になる。
**土台 PR に含める**ことで、Sprint 1 の 3 US すべてが初日から TDD を開始できる。

### 決定 3: 各 US は `main` 起点 + `mvp` からのファイル単位ポート

Sprint 1 の 3 ブランチはいずれも `main`(土台マージ後)から分岐し、該当ファイルだけを `mvp` から取り込んで TDD 改修する。

#### US-005 `feature/US-005-admin-auth`(採用分類 1: そのまま採用可能)

- 作成元: `main`(土台マージ後)
- 起点ファイル(`git checkout mvp -- ...`): `lib/auth.ts`, `middleware.ts`, `lib/auth/session.ts`,
  `app/admin/login/**`, 認可集約ヘルパー(`requireAdminSession` 相当), `app/api/auth/[...nextauth]/route.ts`
- 改修範囲: 分類 1 のため機能改変は最小。**認証・認可の必須テスト追加が主作業**。仮実装・デバッグコードの除去確認。
- TDD で追加すべきテスト(architect.md 必須対象=認証・認可):
  - `authorize()` 資格情報検証: 正当/誤り/存在しないユーザー、bcrypt 照合失敗 → セッション未確立
  - セッションガード(`requireAdminSession`): 未認証 → 拒否、認証済み → 通過
  - middleware: 未認証で `/admin/**` → `/admin/login` リダイレクト、公開ルート(`/`, `/reserve/**`, `/api/public/**`)は素通し
  - Server Action 直接呼び出し: 未認証 → UNAUTHORIZED(ページ保護に依存せず Action 先頭ガードで拒否)
- 完了条件: 受入条件(US-005 全 AC)充足 / 認証・認可テスト成功 / lint・typecheck 成功 / 秘密情報・デバッグコードなし /
  MVP 専用仮実装が残っていない。

#### US-001 `feature/US-001-consultation-select`(採用分類 2: リファクタ後採用)

- 作成元: `main`(土台マージ後)
- 起点ファイル: `app/reserve/steps/StepConsultation.tsx`, および予約ウィザードのシェル/ルーティング・下書き永続化基盤
  (`app/reserve/**` のうち US-001 に必要な入口部分)
- 改修範囲: 分類 2 のため**必ずリファクタ**。ウィザードの骨格・下書き永続化を US-001 の責務範囲で確立
  (日時ステップ本体=StepDateTime は US-002 で追加)。
- TDD で追加すべきテスト(必須対象外だがプロダクトフェーズは TDD 必須):
  - typeId → 所要時間マッピング(0→90 / 1→60 / 2→30 分)の純粋関数
  - 拠点/来店経験の変更時に既選択の日時が破棄され条件を取り直す(下書きリセット)
  - 未選択で次へ進めない(バリデーション)
- 完了条件: US-001 全 AC 充足 / 上記ロジックテスト成功 / lint・typecheck 成功 / 暫定文言(確認7)はプレースホルダで進行し
  本番差し替え可能な形 / 仮実装・秘密情報なし。

#### US-002 `feature/US-002-availability-calendar`(採用分類 2: リファクタ後採用)

- 作成元: `main`(**US-001 マージ後**。理由は下記「実行順序」)
- 起点ファイル: `lib/reservation/availability.ts`, `lib/reservation/judge.ts`,
  `app/reserve/steps/StepDateTime.tsx`, 公開空き状況取得の Route Handler(GET 系)
- 改修範囲: 分類 2 のため**必ずリファクタ**。判定ロジックを純粋関数として維持しつつテスト可能性を担保。
- TDD で追加すべきテスト(**空き状況判定の正しさ=データ整合性隣接領域として必須扱い**):
  - 3 段階しきい値: 残数 0→不可 / 1-3→残りわずか(△) / 4 以上→予約可能(○)
  - 複数枠またぎ: 90 分は連続 3 枠すべて空きで初めて可、1 枠でも満枠・休憩・営業時間外なら不可
  - ラストオーダー境界: 日向 5 時間前 / 延岡 1.5 時間前 を過ぎた枠は不可(境界値テスト)
  - 祝日・終日休診・時間帯不定休 → 不可
  - **JST↔UTC 範囲取得**: 当日〜21 日先の 30 分刻み、端の枠を取りこぼさない「広め取り込み」ロジックの検証
    (user-stories.md が明示する検証テスト)
  - 日付ページング: 別期間で対応する空き状況を取得
- 完了条件: US-002 全 AC 充足 / 上記判定・範囲テスト成功 / lint・typecheck 成功 / 残数算出仕様(確認1)は
  暫定進行しつつしきい値は表示層で後追い調整可能な形 / 仮実装・秘密情報なし。

### 決定 4: `mvp` ブランチは今後リファレンス専用(凍結)

Sprint 開始以降、`mvp/feature/reservation-mvp` へは新規開発を行わない。読み取り専用の分類元として保持する。
これにより「ポート元が動く」状態が保たれ、分岐差異の発生を防ぐ。

## 実行順序

```
[土台] chore/adopt-mvp-scaffold  ──(main へマージ)── ★全 US の前提・最優先
                                     │
              ┌──────────────────────┴──────────────────────┐
       (並行可) feature/US-005-admin-auth          feature/US-001-consultation-select
              管理系(認証ドメイン)                  利用者フロー入口(公開ドメイン)
                                                              │(利用者フローは直列依存)
                                                              ▼
                                                 feature/US-002-availability-calendar
                                                  (US-001 マージ後に着手)
```

- **土台 PR を最初に単独で `main` へマージ**(ブロッカー。これ無しに US 着手不可)。
- **US-005 と US-001 は並行可能**。ドメインが分離(管理系の認証 vs 公開の予約入口)し、触るファイルが重複しないため。
  US-005 は middleware で `/admin/**` のみ保護し公開ルートを素通しするため、US-001 の公開画面と衝突しない。
- **US-002 は US-001 に直列依存**(sprint-plan の利用者フロー US-001→US-002)。US-002 は拠点・来店経験(typeId/所要時間)
  が確定した状態を前提とし、US-001 が確立するウィザード骨格・下書きの上に日時ステップを積む。
  US-001 マージ後に `main` から分岐して着手する(競合と手戻りを最小化)。
- US-002 の空き状況は US-008/009/010 の設定を反映するが、判定ロジックは seed データでテスト可能なため
  US-008/009/010(Sprint 3/4)を待たない(ソフト依存)。エンドツーエンド整合確認は US-010 完了後。

## リスク

- **schema 突合リスク**: `main` の `prisma/schema.prisma` と `mvp` のスキーマ/マイグレーションが乖離している可能性。
  土台 PR で MVP のマイグレーションを正とし差分を確認・確定する(土台 PR の明示タスク)。未確定のまま US を進めない。
- **分類 2 の「無改修コピー」混入**: `git checkout mvp -- ...` は MVP コードをそのまま取り込む。分類 2/3 のファイルを
  リファクタ/再実装せずマージすると原則違反。レビューで「分類 2 が MVP のまま入っていないか」を必ず確認する。
- **MVP 専用仮実装の紛れ込み**: インメモリ・レート制限、同期メール、店舗電話番号プレースホルダ等。Sprint 1 の 3 US は
  これらに触れない(mail/rateLimit を持ち込まない)ため低リスク。ただし取り込みファイル範囲を US スコープに厳密限定する。
- **US-001/US-002 の `app/reserve/**` 共有ファイル競合**: 直列化(US-002 は US-001 マージ後)で回避。並行させないこと。
- **テスト不在からの立ち上げコスト**: 土台 PR にテストハーネスを集約することで一度きりに抑える。各 US は
  受入条件由来のテストのみに集中できる。
- **確認1(残数算出方法)未解決**: US-002 は暫定進行可。ただし「不可」判定時は残数仕様の軽微な手戻りが発生し得るため
  早期のクライアント確認を推奨(sprint-plan の指摘どおり)。しきい値・表示は表示層で吸収できる設計を保つ。

---

## 未決事項

- 土台 PR の Prisma スキーマ最終確定(main と MVP の突合結果)。突合完了までは US 着手を保留する。
- Vitest 以外にコンポーネントテスト(React Testing Library 等)や E2E(Playwright 等)を Sprint 1 で導入するか。
  本 ADR では**単体・ロジックテスト(Vitest)を土台で立ち上げる**のみを確定し、E2E は後続スプリントで判断する。
- CI(GitHub Actions 等)でマージ条件(test/lint/typecheck)を自動ゲート化するかは US-015(インフラ)で確定。
  Sprint 1 ではローカル + PR レビューでマージ条件を担保する。
- US-001 の来店経験の選択肢文言(確認7)、US-002 の残数算出許容性(確認1)はクライアント確認待ち。暫定進行で吸収する。
